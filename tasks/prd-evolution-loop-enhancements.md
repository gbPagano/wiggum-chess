# PRD: Evolution Loop Enhancements

## 1. Introduction / Overview

Extend the Wiggum evolution loop so it behaves more like a production-ready autonomous harness instead of a bare orchestration script. The updated loop must provide clearer operator visibility, consume optional user-provided idea queues during hypothesis selection, manage engine versions as stored release artifacts rather than git refs, and make more flexible promotion decisions when match results are inconclusive but show positive signals.

The goal is to make the loop easier to operate unattended, easier to audit, and better aligned with versioned engine artifacts stored in `chess-engine/versions/`.

## 2. Goals

- Hide the OpenClaude banner and intermediate raw CLI output during phase execution.
- Show concise, operator-friendly phase progress for each iteration.
- Allow the propose phase to consume optional ideas from a user-supplied checklist file.
- Let the decision phase use an additional Stockfish comparison when candidate-vs-baseline evidence is inconclusive.
- Permit promotion of inconclusive candidates only when there is still a positive signal after the additional comparison.
- Version candidate engines using stored engine artifacts in `chess-engine/versions/<tag>/` instead of git refs.
- Persist accepted engine binaries inside `chess-engine/versions/<tag>/` and use those stored artifacts as the next baseline.
- Update the baseline version automatically after an accepted iteration so the next iteration starts from the promoted engine version.

## 3. User Stories

### US-001: Suppress raw OpenClaude CLI output
**Description:** As an operator, I want the evolution loop to suppress the OpenClaude banner and raw intermediate phase output so that the terminal stays readable during unattended runs.

**Acceptance Criteria:**
- [ ] The orchestration script does not print the OpenClaude banner during propose, implement, benchmark, or decide phases.
- [ ] The orchestration script does not stream raw skill output to the terminal by default.
- [ ] The default output behavior matches `scripts/ralph/ralph.sh`: concise terminal progress with captured underlying tool output.
- [ ] Each phase still captures its stdout/stderr for later inspection in session artifacts or logs.
- [ ] The final per-phase outcome remains available in iteration artifacts.
- [ ] `bash -n scripts/evolution-loop.sh` passes.

### US-002: Show current and remaining phases per iteration
**Description:** As an operator, I want the loop to show the current phase and the remaining phases for the active iteration so that I can tell whether the run is progressing or stuck.

**Acceptance Criteria:**
- [ ] At the start of each iteration, the script prints the ordered phase list for that iteration.
- [ ] Before each phase begins, the script prints the current phase name.
- [ ] Before each phase begins, the script prints which phases remain in the current iteration.
- [ ] Output is concise and does not include raw model banner text.
- [ ] `bash -n scripts/evolution-loop.sh` passes.

### US-003: Accept optional ideas file for proposing
**Description:** As an operator, I want to pass a file containing possible engine ideas so that the propose phase can prefer curated ideas generated by me or other agents.

**Acceptance Criteria:**
- [ ] The script accepts a new optional argument for an ideas file path.
- [ ] If the ideas file is omitted, the propose phase behaves exactly as it does today.
- [ ] If the ideas file exists but is empty, the propose phase ignores it and behaves like no file was provided.
- [ ] The supported format is Markdown checklist entries using `- [ ] idea text`.
- [ ] The propose phase can read pending ideas from that file during hypothesis selection.
- [ ] The selected hypothesis records whether it came from the ideas file or from the model's own proposal.
- [ ] `bash -n scripts/evolution-loop.sh` passes.

### US-004: Track proposal source in iteration metadata
**Description:** As a reviewer, I want each iteration to record whether the hypothesis came from the user ideas file or from the model itself so that versioning and audit behavior can follow the correct policy.

**Acceptance Criteria:**
- [ ] `iteration.json` stores the proposal source as either `user_ideas_file` or `self_proposed`.
- [ ] `hypothesis.md` states the source of the selected idea.
- [ ] The implementation and decision phases can read the proposal source from `iteration.json`.
- [ ] Existing iterations without this field fail clearly or default through an explicit compatibility rule documented in the PRD.
- [ ] `bash -n scripts/evolution-loop.sh` passes.

### US-005: Bump UCI version during implementation
**Description:** As an operator, I want the implementation phase to bump the engine version shown in the UCI name so that each candidate can be distinguished during benchmarking.

**Acceptance Criteria:**
- [ ] The implement phase updates the engine version string exposed in the UCI name for the candidate build.
- [ ] If the hypothesis source is `self_proposed`, the candidate version increments the minor version from the current baseline (example: `v0.1` → `v0.2`).
- [ ] If the hypothesis source is `user_ideas_file`, the candidate version increments the major version from the current baseline (example: `v0.1` → `v1.0`).
- [ ] The candidate version is recorded in `iteration.json` and `implementation.md`.
- [ ] The baseline engine version remains unchanged until the decision is `accepted`.
- [ ] Candidate and baseline UCI names are distinguishable during benchmark runs.
- [ ] The implementation phase produces a candidate binary named `wiggum-engine` for later persistence into version artifacts.

### US-006: Resolve baseline from stored version artifacts
**Description:** As an operator, I want the loop to use stored engine versions from `chess-engine/versions/` instead of git refs so that baseline selection is explicit and reproducible from release artifacts.

**Acceptance Criteria:**
- [ ] The session accepts a baseline version label that resolves to `chess-engine/versions/<tag>/`.
- [ ] The baseline version directory is required to contain a compiled engine binary named `wiggum-engine` and no other mandatory engine artifact.
- [ ] The loop fails fast with a clear error if the requested baseline version directory is missing or if `wiggum-engine` is absent.
- [ ] `session.env` records the selected baseline version path instead of a git ref.
- [ ] `iteration.json` records the resolved baseline version path used for that iteration.
- [ ] The benchmark phase can identify the baseline engine binary from the stored version directory.

### US-007: Persist accepted engine artifacts as new versions
**Description:** As an operator, I want accepted candidates to be saved as versioned engine artifacts so that future iterations and external benchmarking use the promoted binary directly.

**Acceptance Criteria:**
- [ ] When an iteration is accepted, the script creates a new directory `chess-engine/versions/<new-version>/`.
- [ ] The accepted candidate's compiled engine binary is copied into that version directory with the filename `wiggum-engine`.
- [ ] The version directory does not require preserving the Cargo `target/` layout.
- [ ] The stored binary path is recorded in session metadata for future iterations.
- [ ] The script does not depend on a git commit ref to resolve the next baseline.

### US-008: Update baseline automatically after promotion
**Description:** As an operator, I want the loop to update the active baseline after a candidate is accepted so that the next iteration evolves from the latest promoted model.

**Acceptance Criteria:**
- [ ] After an `accepted` decision, the session metadata updates the active baseline version to the new promoted version.
- [ ] After an `accepted` decision, the next iteration uses the stored binary and metadata from the new version directory.
- [ ] After a `rejected`, `failed`, or `inconclusive` decision, the active baseline remains unchanged.
- [ ] The final session summary lists all accepted baseline promotions in order.
- [ ] `bash -n scripts/evolution-loop.sh` passes.

### US-009: Run extra Stockfish comparison for inconclusive results
**Description:** As an operator, I want the decision flow to trigger an extra comparison against Stockfish when candidate-vs-baseline results are inconclusive so that promotion decisions use more context than a single neutral match.

**Acceptance Criteria:**
- [ ] If the baseline-vs-candidate benchmark result is inconclusive, the decision flow can request or run an additional Stockfish comparison.
- [ ] The comparison uses the prior stored baseline benchmark result against Stockfish when available.
- [ ] The candidate's Stockfish result is recorded in `benchmark.md` or a linked decision artifact.
- [ ] The decision artifact explicitly states whether the Stockfish comparison changed the recommendation.
- [ ] If the Stockfish comparison cannot run, the iteration records that limitation instead of silently ignoring it.

### US-010: Allow positive-signal promotion after inconclusive evidence
**Description:** As an operator, I want the decision phase to promote an inconclusive candidate when there is still a positive signal after extra analysis so that potentially stronger engines are not discarded too aggressively.

**Acceptance Criteria:**
- [ ] If the direct candidate-vs-baseline result is inconclusive, the decision logic performs the additional Stockfish comparison before making the final call.
- [ ] If the Stockfish comparison is also inconclusive, the decision logic explicitly evaluates whether there is any positive signal suggesting likely improvement.
- [ ] Positive-signal criteria are written to the PRD and represented in `decision.md` when used.
- [ ] If no positive signal exists, the candidate is not promoted.
- [ ] If a positive signal exists and the candidate is promoted, `decision.md` explains why promotion was allowed despite inconclusive match evidence.
- [ ] `iteration.json` records the exact evidence used for the promotion decision.

**Positive-signal criteria:**
- The positive-signal path is only considered after both the direct candidate-vs-baseline benchmark and the extra Stockfish comparison remain inconclusive.
- Promotion remains disallowed by default when `benchmark.sufficientForPromotion` is `false`; the only exception is an explicit positive-signal judgment recorded in the iteration artifacts.
- Any positive signal must be written into `iteration.json.stockfishComparison.positiveSignal` with whether it was evaluated, whether it is present, a short summary, and the exact supporting evidence list.
- If that positive signal is used to justify promotion, `decision.md` must explain why the evidence still suggests likely improvement despite inconclusive match results and repeat the exact supporting evidence.
- If the positive-signal evidence is absent or not persuasive enough to document explicitly, the outcome must stay non-promoted (`rejected` or `inconclusive`).

## 4. Functional Requirements

- FR-1: The orchestration script must run skills without printing the OpenClaude banner or raw live model output to the terminal by default.
- FR-2: The orchestration script must capture phase output into deterministic logs or artifacts that can be inspected after execution.
- FR-3: The orchestration script must print concise phase progress showing the current phase and the remaining phases for the active iteration.
- FR-4: The orchestration script must accept an optional ideas-file argument.
- FR-5: The ideas file format must be Markdown checklist entries using `- [ ] ...` syntax.
- FR-6: The propose phase must ignore the ideas file when the file is missing, omitted, or contains no pending checklist items.
- FR-7: The propose phase must record whether the chosen hypothesis came from the user ideas file or from self-proposal.
- FR-8: The implementation phase must compute the candidate engine version from the active baseline version and the proposal source.
- FR-9: Self-proposed ideas must increment the minor version only (example: `v0.1` → `v0.2`).
- FR-10: Ideas selected from the user ideas file must increment the major version only (example: `v0.1` → `v1.0`).
- FR-11: The implementation phase must update the UCI-visible engine version string for the candidate build before benchmarking.
- FR-12: The baseline engine must be resolved from `chess-engine/versions/<tag>/` rather than from git commit refs.
- FR-13: Each version directory under `chess-engine/versions/<tag>/` must store exactly one mandatory engine artifact: a compiled binary named `wiggum-engine`.
- FR-14: The benchmark phase must run the candidate binary against the baseline binary resolved from the version directory.
- FR-15: The benchmark and decision artifacts must record the exact candidate binary path and baseline binary path used.
- FR-16: If a candidate-vs-baseline result is inconclusive, the decision flow must perform or request an additional Stockfish comparison for the candidate and compare it with the stored baseline-vs-Stockfish result when available.
- FR-17: If the Stockfish comparison is also inconclusive, the decision flow must evaluate whether any positive signal remains.
- FR-18: The agent may decide the positive-signal heuristic case by case, but the rationale must be explicit and auditable in `decision.md` and `iteration.json`.
- FR-19: If a candidate is accepted, the orchestration script must create a new version directory under `chess-engine/versions/<promoted-version>/` and copy the compiled candidate binary there as `wiggum-engine`.
- FR-20: If a candidate is accepted, the session metadata must update the active baseline version so the next iteration starts from the promoted version directory.
- FR-21: If a candidate is rejected, failed, or remains inconclusive without positive-signal promotion, the active baseline version must remain unchanged.
- FR-22: The session summary must list baseline promotions and the version directory for each accepted engine.
- FR-23: After an idea from the user-supplied checklist file is selected and tested, the system must mark that checklist item as used.

## 5. Non-Goals (Out of Scope)

- Building a general-purpose release-management system beyond the evolution loop.
- Designing a UI or dashboard for loop execution.
- Supporting multiple ideas-file formats beyond Markdown checklist syntax in this iteration.
- Rewriting the benchmark engine or chess-runner match protocol.
- Adding distributed execution across multiple machines.
- Supporting arbitrary semver bump policies beyond the explicitly requested minor/major rules.

## 6. Design Considerations

- Terminal output should stay human-readable during long unattended runs.
- The operator should always be able to tell whether the current iteration is proposing, implementing, validating, benchmarking, or deciding.
- Session and iteration artifacts should remain the source of truth for detailed evidence, while terminal output stays concise.
- The ideas file should be easy for humans and other agents to append to without special tooling.

## 7. Technical Considerations

- The current loop resolves the accepted baseline using git refs; this must be replaced with version-directory artifact resolution.
- The current benchmark artifacts already distinguish candidate and baseline conceptually, but the new flow must record explicit binary paths from `chess-engine/versions/`.
- Persisted version directories should contain the promoted binary named `wiggum-engine`; no other engine artifact is mandatory.
- Version parsing and bumping rules must be deterministic and reject invalid version strings early.
- The implementation phase must know where the engine's UCI name is defined so it can update the candidate version before benchmark runs.
- The decision flow may let the agent decide what counts as a positive signal case by case, but that reasoning must be written explicitly in the artifacts.
- The system should handle the case where a previous baseline version lacks Stockfish comparison data by recording that the extra comparison could not be made.
- The default operator output should behave like `scripts/ralph/ralph.sh`, with concise progress in the terminal and captured underlying model output available separately.
- After an idea from the checklist file is tested, the system should mark that checklist entry so it is not proposed again automatically.

## 8. Success Metrics

- Operators can run the loop without seeing raw OpenClaude banner noise.
- Operators can identify the current iteration phase from terminal output immediately.
- Every accepted candidate produces a reusable version directory in `chess-engine/versions/`.
- Future iterations can start from a stored version artifact without depending on git refs.
- Inconclusive runs produce explicit, auditable decision reasoning instead of silent fallback behavior.
- User-supplied idea queues can influence proposing without breaking autonomous fallback behavior.

## 9. Resolved Decisions

- Each `chess-engine/versions/<tag>/` directory must require only one mandatory engine artifact: a compiled binary named `wiggum-engine`.
- When both direct and Stockfish comparisons remain inconclusive, the agent may decide case by case whether a positive signal exists, but it must explain that reasoning explicitly in `decision.md` and `iteration.json`.
- After a checklist idea is selected and tested, the system should mark that entry as used in the ideas file.
- The loop should support an optional verbose mode for debugging suppressed skill output.
- The default terminal behavior should match `scripts/ralph/ralph.sh`: concise progress in the terminal, with underlying model output captured separately.
- Accepted version directories should store only the promoted binary `wiggum-engine`; preserving the Cargo `target/` layout is out of scope.
